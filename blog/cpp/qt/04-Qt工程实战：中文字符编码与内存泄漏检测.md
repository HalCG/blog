# Qt 工程实战：中文字符编码与内存泄漏检测

在进行 Qt 项目开发时，有两个最常见也最棘手的工程问题常常阻碍项目上线：**中文字符界面乱码**，以及**隐蔽的内存泄漏（Memory Leak）**。

本篇将从实战出发，为你提供一击必杀的中文乱码解决方案，并介绍 Qt 环境下内存泄漏的防范技巧与主流检测工具。

---

## 1. 彻底解决 Qt 中文乱码与字符编码冲突

在 Windows 平台使用 MSVC 编译器开发 Qt 项目时，经常会遇到界面上的中文汉字显示为“锟斤拷”等奇怪乱码的问题。

### 1.1 乱码产生的本质原因
这主要是由于**源码文件编码**、**编译器执行字符集**与 **Qt 内部编码**三者不一致导致的：
1. **源码编码**：现代编辑器（如 VSCode、Qt Creator）默认以 **UTF-8** 保存源码文件。
2. **编译器执行字符集**：Windows 下的 MSVC 编译器在默认情况下，会采用系统本地编码（中国地区默认为 **GBK**）来解析和编译源文件中的字符。
3. **Qt 内部编码**：`QString` 内部统一使用 **UTF-16** 进行存储和显示。在接收字符串字面量时，Qt 默认其为 UTF-8。

因为编译器用 GBK 去强行解析 UTF-8 编码的汉字，编译出来的二进制数据本身就是错的，显示出来自然是乱码。

---

### 1.2 解决方案（三步走）

#### 第一步：强制编译器采用 UTF-8 编译（最关键）
无论你在代码中怎么进行格式转换，如果编译器编译出来的字符已经是错的，就没有任何意义。因此，必须在构建配置中加入 `/utf-8` 编译参数。

* **如果使用 CMake 构建项目**，在 `CMakeLists.txt` 中添加：
  ```cmake
  if(MSVC)
      add_compile_options(/utf-8) # 强制 MSVC 使用 UTF-8 解析和执行字符集
  endif()
  ```
* **如果使用 qmake 构建项目**，在 `.pro` 文件中添加：
  ```qmake
  msvc {
      QMAKE_CFLAGS += /utf-8
      QMAKE_CXXFLAGS += /utf-8
  }
  ```

#### 第二步：在代码中规范使用 `QStringLiteral`
不要直接写 `QString("中文")`，因为这会在运行期产生临时的解析转换开销。推荐使用 `QStringLiteral` 宏。
* **原理**：`QStringLiteral` 会在**编译期**直接生成一个存储在只读数据段的 `QString` 内部只读对象。不仅完美避开了运行期字符集转换导致的乱码，还极大地提升了执行效率。
  ```cpp
  ui->label->setText(QStringLiteral("欢迎使用本系统"));
  ```

#### 第三步：读取本地外部非 UTF-8 文本
如果需要读取本地以 GBK/ANSI 格式保存的日志或配置文件，应使用 `QString::fromLocal8Bit`：
  ```cpp
  QString local_str = QString::fromLocal8Bit(raw_char_array);
  ```

## 2. Qt 内存泄漏检测与安全防范

尽管 Qt 的 QObject 对象树在父对象释放时会自动销毁子对象，但以下场景仍然会产生内存泄漏：
1. 创建对象时没有指定 `parent`，且没有手动 `delete`。
2. 将非 QObject 对象（如自定义的 C++ 结构体、数据缓存）分配在堆上，却漏写了释放代码。
3. 线程对象在运行结束时没有被正确回收。

---

### 2.1 现代 C++ 安全防范：Qt 专属智能指针
为了防止手动释放遗漏，应善用 Qt 提供的智能指针：

#### 2.1.1 QPointer (悬空指针保护哨)
`QPointer``<T>``` 是专为 `QObject` 设计的**弱引用护卫指针**。
* **解决痛点**：当它指向的 `QObject` 被其他地方销毁（`delete`）后，`QPointer` 会**自动被置为 `nullptr`**。这能完美避免指针成为悬空指针，防止二次释放或非法野指针调用崩溃。
```cpp
QPointer<QLabel> label = new QLabel("Status");
// ... 
delete label; // 销毁对象
if (label.isNull()) {
    // 💡 自动被置为了 nullptr，判断为真。安全！
}
```

#### 2.1.2 QScopedPointer (作用域指针)
类似于标准库的 `std::unique_ptr`。当对象离开当前大括号作用域时，自动调用析构函数释放堆内存。
```cpp
void process() {
    QScopedPointer<MyData> data(new MyData());
    data->process();
} // 离开作用域，data 自动释放
```

---

### 2.2 内存泄漏检测工具推荐

#### 2.2.1 Windows 开发环境：VLD (Visual Leak Detector)
VLD 是一款针对 MSVC 编译器的免费、高效的内存泄漏检测工具。它能完美集成在 Qt Creator 或 Visual Studio 中。
* **使用方法**：
  1. 下载安装 VLD。
  2. 在 Qt 项目的 `main.cpp` 头部引入头文件即可（无需修改任何业务代码）：
     ```cpp
     #include <vld.h> // 仅需引入此头文件
     ```
  3. 当程序退出时，VLD 会在调试输出（Application Output）窗口打印出所有未释放内存的详细堆栈信息，甚至能精确指出**漏写 delete 的是哪一个文件的哪一行代码**。

#### 2.2.2 Linux 开发环境：Valgrind
在 Linux 下，Valgrind 是检测 C/C++ 内存泄漏的行业标准工具。
* **命令行执行**：
  ```bash
  valgrind --tool=memcheck --leak-check=full ./my_qt_app
  ```
* 它会全面监控程序的所有内存申请与归还操作，并在程序退出时输出完整的内存泄漏报告。
