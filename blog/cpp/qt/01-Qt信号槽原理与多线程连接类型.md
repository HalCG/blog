# Qt 信号槽原理与多线程连接类型

信号与槽（Signals & Slots）是 Qt 框架的核心基石，它提供了一种**类型安全、松散耦合**的对象间通信机制。在多线程编程中，合理配置信号槽的连接类型，是保证跨线程调用数据安全与同步的关键。

本篇将深入剖析 Qt 信号槽的编译与执行机制，并重点详解 5 种核心的**连接类型（Qt::ConnectionType）**。

---

## 1. 信号与槽的底层机制：MOC 与元对象系统

在 C++ 编译期，标准的编译器并不能识别 `signals`、`slots` 或 `emit` 这一类 Qt 专属关键字。Qt 是通过 **MOC (Meta-Object Compiler，元对象编译器)** 来支持这些特性的。

```
                    MOC 扫描头文件 (含有 Q_OBJECT)
                          │
                          ▼
 ┌──────────────┐   ┌─────────────┐   ┌──────────────────────────┐
 │  widget.h    ├──►│  moc_widget.cpp  ├──► 合并送入 C++ 编译器编译
 └──────────────┘   └─────────────┘   └──────────────────────────┘
```

### MOC 做的事情：
1. **扫描头文件**：编译开始前，MOC 会扫描项目头文件，寻找包含 `Q_OBJECT` 宏的类。
2. **生成元对象代码**：为该类自动生成一个名为 `moc_xxx.cpp` 的源文件。
3. **实现信号函数**：信号在头文件中只需声明，无需编写具体的 C++ 实现代码，因为 **MOC 会在 `moc_xxx.cpp` 中自动生成信号函数的具体实现**（用于打包参数并通知连接的槽函数）。
4. **元数据映射**：生成保存类名、信号与槽函数名及其参数索引的字符串数组，从而在运行期支持**反射（Reflection）**与动态函数定位。

## 2. 深入理解 5 种连接类型（ConnectionType）

在使用 `QObject::connect()` 时，第五个参数决定了信号发出后，槽函数在哪个线程、以何种形式被执行：

```cpp
QObject::connect(sender, &Sender::sig, receiver, &Receiver::slot, Qt::AutoConnection);
```

### 2.1 Qt::AutoConnection（自动连接 - 默认值）
* **规则**：如果 **发送者信号发出的线程** 与 **接收者对象所属的线程（即 `receiver->thread()`）** 相同，则自动转换为 `DirectConnection`；如果不同，则自动转换为 `QueuedConnection`。
* **适用场景**：绝大多数常规开发场景。

### 2.2 Qt::DirectConnection（直接连接）
* **规则**：信号发出后，槽函数在**发送信号的线程**中直接执行（同步调用）。
* **本质**：等同于普通的成员函数回调。
* **⚠️ 风险**：如果接收者对象在另一个线程中运行，而它的槽函数却在发送者线程中执行，如果槽函数内部修改了接收者的成员变量，将导致多线程竞态读写，**极其不安全**！

### 2.3 Qt::QueuedConnection（队列连接）
* **规则**：信号发出后，Qt 将该调用打包为一个事件（Event），投递到**接收者线程的事件循环队列**中。接收者线程在下一次轮询到该事件时，在其本线程中执行该槽函数（异步调用）。
* **本质**：跨线程安全通信的标准方式。
* **安全保障**：确保了接收者对象的所有操作（槽函数）都在接收者自己的线程内顺序执行，避免了跨线程冲突。

### 2.4 Qt::BlockingQueuedConnection（阻塞队列连接）
* **规则**：与 `QueuedConnection` 类似，槽函数在接收者线程中执行。但**发送者线程会立即陷入阻塞**，直到接收者线程执行完该槽函数后，发送者线程才会被唤醒并继续执行。
* **⚠️ 致命死锁陷阱**：如果发送者与接收者在**同一个线程**，使用此连接类型会导致该线程自己等待自己执行，直接发生**永久性死锁**！

### 2.5 Qt::UniqueConnection（唯一连接）
* **规则**：这是一个辅助标志，可以与其他类型通过按位或 `|` 组合使用。它能确保相同的信号和槽之间只建立一个连接。如果重复连接，则会失败并返回 `false`，有效防止了重复连接导致槽函数被执行多次的问题。

## 3. ⚠️ 跨线程队列连接避坑：注册自定义数据类型

当你使用 `QueuedConnection`（包括多线程下的 `AutoConnection`）跨线程发送自定义类型的参数（如自定义结构体 `MyStruct`）时，程序可能会抛出如下运行时错误，且槽函数无法被触发：
> `QRegisterMetaType: Unable to register parameter type 'MyStruct'`

### 3.1 原因分析
因为队列连接需要将信号的参数复制一份存入事件队列中，稍后再由接收者线程取出。Qt 默认只认识内置的基本类型（如 `int`、`double`、`QString` 等）。对于自定义的类或结构体，Qt 不知道如何拷贝它。

### 3.2 解决方案
在发起连接前，必须使用 `qRegisterMetaType` 注册该类型，使其加入 Qt 的元对象类型系统中：

```cpp
#include <QMetaType>

struct MyStruct {
    int id;
    QString name;
};

// 💡 必须在 QObject::connect() 执行前注册
qRegisterMetaType<MyStruct>("MyStruct");

// 之后即可安全地在跨线程信号槽中传递 MyStruct 对象
connect(thread, &WorkerThread::dataReady, this, &MainWindow::updateData);
```
