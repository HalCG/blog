# TCP 与 UDP 协议核心对比及 I/O 多路复用（select-epoll）机制

在构建高性能 C++ 网络服务（如 Web 服务器、RPC 框架、游戏网关）时，传输层协议的选型与操作系统的 I/O 事件分发模型是架构设计的核心。

本篇将对比 **TCP 与 UDP 协议** 的物理差异，详解 **TCP 粘包与拆包** 解决方案及 **拥塞控制** 四大阶段，并深度剖析 **select 与 epoll** 多路复用机制的底层实现及 LT/ET 触发模式。

---

## 1. TCP 与 UDP 的核心特性对比

| 特性维度 | TCP (Transmission Control Protocol) | UDP (User Datagram Protocol) |
| :--- | :--- | :--- |
| **连接机制** | **面向连接**（收发数据前必须三次握手） | **无连接**（直接知道目的 IP 和端口即可发送） |
| **可靠性** | **可靠交付**（无差错、不丢失、不重复、按序到达） | **不可靠**（尽最大努力交付，无确认和重传机制） |
| **数据边界** | **面向字节流**（数据如流水，无物理边界） | **面向报文**（保留应用层发送的数据包物理边界） |
| **首部开销** | 较重（固定 **20 字节**，最大可达 60 字节） | 极轻（固定 **8 字节**） |
| **通信模式** | 仅支持**点对点**（一对一） | 支持单播、**广播**和**组播**（一对多、多对多） |
| **适用场景** | 网页浏览（HTTP）、文件传输（FTP）、邮件（SMTP） | 实时音视频通话、在线游戏、DNS 查询 |

## 2. 经典工程问题：TCP 粘包与拆包

“粘包”和“拆包”是 TCP 面向字节流特性带来的必然现象。

```
  发送端发送:   [ Packet A ]   [ Packet B ]
  
  接收端可能收到:
  情况一 (粘包):  [ Packet A  Packet B ]          <-- 两包融合为一个字节流被取走
  情况二 (拆包):  [ Packet A_part1 ] [ A_part2 B ] <-- A包被拆分，部分与B包融合
```

### 2.1 产生原因
1. **TCP 面向字节流**：TCP 并不理解应用层的业务数据逻辑。它只把所有数据看作一串无结构的字节流，发送时会根据网卡 MTU 大小进行拆分发送。
2. **Nagle 算法优化**：为了减少网络小包，TCP 默认会开启 Nagle 算法，将多个微小的数据包合并为一个大的数据段一次性发送，直接造成粘包。
3. **💡 为什么 UDP 没有粘包问题？**：因为 UDP 是面向报文的。UDP 协议头中有明确的“长度”字段，每个 UDP 包在网络层都被独立打包发送，接收端每次 `recvfrom` 只能刚好读取一个完整的 UDP 报文，因此 **UDP 天然不存在粘包/拆包问题**。

### 2.2 解决方案
在应用层设计**协议边界**：
* **定长分包**：规定每个业务包固定为 $N$ 字节。不足的用空格或 `\0` 填充（适用于心跳包等简单场景）。
* **特殊分隔符**：在每个包末尾追加特定字符（如 `\r\n`）。接收端循环读取直到遇到分隔符进行切分。
* **自定义包头（TLV 模式 - 最常用）**：在业务包头部增加固定长度的字段，专门存储**包体数据的实际长度**。接收端先读取固定长度的包头获取长度 $L$，再精确 `recv` 长度为 $L$ 的字节流。

## 3. TCP 拥塞控制四大阶段

为了防止过多的数据注入网络，导致中间路由器过载丢包，TCP 引入了拥塞控制算法，动态控制**拥塞窗口（cwnd）**的大小：

```
 cwnd 窗口大小
  ▲
  │                                    / 拥塞发生 (cwnd 减半)
  │                                   /  ssthresh 设为此时的一半
  │                     / 拥塞发生   /
  │                    /  cwnd=1    / ◄── 快速恢复与拥塞避免阶段
  │       / 拥塞避免  /             /
  │      /           /             /
  │     /            │            /
  │    /  ◄── 慢启动  │           /
  │   /              │          /
  └───┴──────────────┴──────────┴─────────────► 时间
```

1. **慢启动（Slow Start）**：连接建立时，将 `cwnd` 初始化为 1 个 MSS（最大报文段长度）。每收到一个 ACK，`cwnd` 呈 **指数级增长**。
2. **拥塞避免（Congestion Avoidance）**：当 `cwnd` 达到慢启动阈值 `ssthresh` 时，为了防止丢包，窗口增长速度降为 **线性增长**（每个往返时间 RTT，窗口只增加 1 个 MSS）。
3. **快速重传（Fast Retransmit）**：接收方如果收到一个失序的报文段，会立即发送重复的 ACK。**一旦发送方连续收到 3 个相同的冗余 ACK**，就认为该报文段已丢失，立即重传该包，而不需要等待该包的超时定时器溢出。
4. **快速恢复（Fast Recovery）**：配合快速重传，当收到 3 个冗余 ACK 时，不将 `cwnd` 降回 1，而是将 `ssthresh` 降为当前窗口的一半，并将 `cwnd` 设为新的 `ssthresh`，直接进入拥塞避免阶段（避免了重新慢启动的低效过程）。

## 4. I/O 多路复用机制：select 与 epoll

I/O 多路复用允许单个线程同时监听成千上万个 Socket 文件描述符（fd）的读写事件，是高性能网络服务的基本支撑。

### 4.1 select 机制与缺陷
* **工作原理**：
  1. 每次调用时，用户态需要将监控的 fd 集合（`fd_set`，底层是位图）拷贝到内核态。
  2. 内核**线性遍历**所有 fd，检查是否有就绪事件。
  3. 如果没有，当前线程休眠；一旦有 fd 就绪，唤醒线程。
  4. 唤醒后，内核将整个 `fd_set` 拷贝回用户态，用户态需要再次**线性遍历** fd 集合找到究竟是哪一个 fd 活跃。
* **三大缺点**：
  1. **并发限制**：受位图大小限制，单个进程默认只能监听 **1024** 个 fd。
  2. **拷贝开销大**：每次调用都要在用户态与内核态之间来回拷贝整个 fd 集合。
  3. **轮询开销大**：查找就绪 fd 的时间复杂度是 $O(N)$，fd 数量多时性能暴跌。

### 4.2 epoll 机制（Linux 特有）
epoll 针对 select 的三大痛点进行了完美优化：
* **三板斧机制**：
  1. **`epoll_create`**：在内核中创建一棵**红黑树**（用于高效存放通过 `epoll_ctl` 注册的所有 Socket fd）和一个**就绪双向链表**（存放发生事件的 fd）。
  2. **`epoll_ctl`**：向红黑树添加/删除 fd。在添加时，会给该 fd 注册一个**事件回调函数**。一旦该 fd 发生事件，硬件中断会自动调用该回调函数，**直接将该 fd 挂入就绪链表**。
  3. **`epoll_wait`**：线程调用后直接检查就绪链表是否为空。如果不为空，直接以 $O(1)$ 的开销返回就绪的 fd，不需要遍历红黑树，也完全不需要在每次调用时重复拷贝所有的 fd 集合。

### 4.3 LT（水平触发）与 ET（边沿触发）模式的区别
* **LT (Level Triggered - 水平触发，默认)**：只要 Socket 的接收缓冲区中**还有数据**，每次调用 `epoll_wait` 都会不断重复通知该 fd。
  * *优点*：编程简单，不易漏读数据。
* **ET (Edge Triggered - 边沿触发)**：只有当 Socket 状态**发生改变**时（例如新数据到达），`epoll_wait` 才会通知一次。即使缓冲区里还有没读完的数据，只要没有新数据来，就不会再次通知。
  * *要求*：必须配合**非阻塞 IO（O_NONBLOCK）**使用，且必须使用循环 `read` 直到返回 `EAGAIN` 错误（表示数据已被全部读完）。
  * *优点*：减少了重复的事件通知次数，适合超高并发服务。
