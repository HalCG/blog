# 第 3 章：基础工具

> 系列：[程序员修炼之道](README.md) · 第 3/9 章

---

## 一句话总结

投资工具链——纯文本、Shell、强力编辑器、版本控制、调试器与文本处理语言——让重复劳动自动化，把脑力留给设计。

---

## 核心观点浓缩

- **纯文本**：配置、数据、文档用可 diff、可脚本处理的格式  
- **Shell 游戏**：组合小命令完成批处理；GUI 点不完的量用脚本  
- **威力编辑**：精通一种编辑器，减少上下文切换  
- **版本控制**：一切源、配置、文档纳入 VCS  
- **调试**：复现、二分、假设验证；别猜  
- **文本处理语言**：awk/sed/Python 等处理日志与批量重构  
- **工程日记**：记录决策、环境、坑，对抗遗忘

---

## 关键概念 / 金句

- 「用纯文本保存知识」——可版本化、可工具化  
- 调试是**科学方法**，不是乱改重试  
- 工具投资一次，复利多年

---

## 本章在全书中的位置

把第 2 章的设计原则**落地到日常环境**。没有 VCS 和调试习惯，第 4、7 章的防御与重构难以执行。

---

## 个人思考与启发

Windows 上 PowerShell / Git / VS 调试器同样适用「基础工具」精神。构建日志、连接诊断输出用文本，便于用户打包上传分析。  
编辑器片段与多光标对大批量改 API 名（如重构 `exportScanData`）省时不亚于写脚本。

---

## 与《软件设计的哲学》对照

- 工具降低**操作复杂度**，设计哲学降低**结构复杂度**——互补  
- 工程日记类似「注释写非显而易见之事」在个人层面的延伸  
- 纯文本配置利于 code review，与战略式编程一致

---

## 重点与注意

> **重点**：**工具链是生产力乘数**，值得专门花时间精通。  
> **重点**：**版本控制**对象应包括配置与文档，不仅是源码。  
> **重点**：调试遵循假设—验证，保存**可复现**最小案例。  
> **注意**：不要陷入无限折腾工具而忘记交付；工具服务于目标。  
> **注意**：威力编辑需练成肌肉记忆，否则收益有限。
